01 — 핵심 결정

두 개의 사이트로 가른다

지금은 세 개의 HTML이 역할 구분 없이 섞여 있습니다. 웹 전환의 첫 결정은 접근 레벨을 셋으로 가르는 것입니다 — 외부 고객, 내부 직원, 그리고 DB를 직접 다루는 관리자. 보는 사람이 다르면 목적·언어·권한·노출 정보가 전부 달라야 합니다.

EXTERNAL · 외부

오퍼링 포탈

고객 · 잠재 리드 — 로그인 없이 공개
  • 목적 — 고객이 스스로 진단하고, 자기 상황에 맞는 오퍼링을 이해
  • 언어 — 고객의 페인포인트 언어. 내부 용어·마진·포컬 이름 없음
  • 흐름 — 진단 → 갭 확인 → 조합 이해 → 리드 남기기
  • 기반 — 지금의 Offering_portal.html
INTERNAL · 내부

인에이블먼트 허브

영업·기술 직원 — 로그인 필요
  • 목적 — 직원이 딜을 설계하고 실행. 판정·급·포컬·공수까지
  • 언어 — 내부 실무 언어. 트랙 판정, 딜 급, HALO 귀속 매출
  • 흐름 — 영업지원 / 기술지원으로 역할별 분리
  • 기반 — presales_hub.html + sales_workflow.html + 허브 v11-0
ADMIN · 관리자

DB에 직접 쓴다

기획·운영 관리자 — 최고 권한 로그인
  • 목적 — 서비스·패키지·포컬·급 기준을 DB에 직접 추가·수정·삭제
  • 권한 — 두 사이트가 읽는 원본 데이터를 쓰는 유일한 곳
  • 반영 — 저장하면 외부·내부 양쪽에 즉시 적용 (재배포 없음)
  • 이력 — 누가 언제 무엇을 바꿨는지 audit_log에 남김
이 허브는 처음부터 짓지 않습니다 — AI Radar를 확장합니다. 기존에 만든 AI Radar(github.com/ISSU-ai/ai-radar)에 이미 Supabase 스키마·인증(JWT·bcrypt·승인제)·관리자 화면·솔루션 카탈로그 35개·pgvector가 구현돼 있습니다. 허브는 그 위에 딜 파이프라인을 얹는 확장입니다. 인증·관리자·RLS를 새로 짜지 않아 개발량이 크게 줄고 검증된 코드에서 시작합니다. 상세는 별첨 통합기획 참조.
왜 셋으로 나누나. 외부 포탈에 딜 급·포컬 담당자·HALO 귀속 매출이 노출되면 사고입니다. 반대로 내부 허브를 고객 언어로 순화하면 직원 도구가 무뎌집니다. 그리고 데이터를 바꾸는 일은 화면을 보는 일과 완전히 다른 권한이라 관리자로 따로 뺍니다. 같은 데이터를 쓰되, 보는 층·언어·쓰기 권한을 분리하는 게 이 기획의 뼈대입니다.
02 — 전체 구조

한 장으로 보는 아키텍처

외부·내부 두 사이트가 하나의 Supabase를 공유합니다. 데이터는 한 곳, 화면은 둘, 권한으로 가릅니다.

세 접근점이 하나의 DB를 공유합니다. 외부·내부는 읽기, 관리자만 원본에 씁니다. Supabase 권한(RLS)이 각 층을 가릅니다.
03 — URL 구조

서브도메인부터 가른다

권한(RLS)만으로 가르지 않습니다. 외부·내부는 URL부터 분리합니다 — 같은 도메인 아래 서브도메인으로. 딜 급·포컬·매출이 외부로 새는 최악을 막는 첫 번째 벽은 권한 규칙이 아니라 애초에 다른 주소입니다.

서브도메인으로 주소를 가르되, 데이터 원본은 하나입니다. 도메인 이름은 예시입니다.

왜 경로(/portal)가 아니라 서브도메인인가

방식예판단
완전 별도 도메인offering.com / hub.내부가장 안전하나 인증서·브랜딩·관리 2배 과함
서브도메인 (선택)offering. / hub. / admin.주소가 갈려 격리되고, 관리는 한 도메인 채택
경로 분리한도메인/portal · /hub한 앱에 공존 → 권한 버그 하나로 유출 위험
서브도메인이 주는 것. ① 외부·내부가 다른 앱으로 분리돼, 내부 코드·데이터가 외부 번들에 아예 안 실립니다. ② 내부·관리자 도메인만 사내망(VPN·IP 제한)으로 묶을 수 있습니다 — 외부에서 주소를 알아도 접근 자체가 막힙니다. ③ 배포·롤백을 각각 독립적으로. DB는 여전히 하나라, 서브도메인이 갈려도 데이터 원본은 공유됩니다.
Supabase 연동 시 준비물. 도메인이 갈리면 각 서브도메인을 Supabase의 허용 목록(CORS·Redirect URL)에 등록해야 합니다. 외부 도메인은 anon key로 공개 읽기 + 리드 쓰기만, 내부·관리자 도메인은 로그인 후 접근. 이건 개발(3단계) 체크리스트에 넣습니다.
04 — 인증

이메일·비밀번호로 시작, SSO는 나중

SSO 연동은 초기 부담이 커서 뒤로 미룹니다. 지금은 이메일/비밀번호로 내부 hub와 admin만 로그인합니다 — 외부 포탈은 로그인 없이 공개. 그리고 회원은 자체 users 테이블로 직접 관리합니다. Supabase Auth에 묶지 않아, 나중에 SSO로 갈아탈 여지를 남깁니다.

외부는 로그인 없음. 내부·admin만 자체 users 테이블로 인증. 비밀번호는 해시로만 저장.
회원 생성

관리자가 발급

회원가입 화면 없음. admin이 users에 계정을 직접 추가. 초기엔 테스트 계정 몇 개로 시작.

역할

admin · sales · tech

role 컬럼으로 구분. admin은 최고 권한(DB 쓰기), 나머지는 hub 사용.

SSO 대비

테이블만 열어둠

나중에 sso_provider·sso_id 컬럼 추가 → 같은 users 테이블로 SSO 흡수.

비밀번호는 절대 평문 저장 안 함 — 타협 불가. users 테이블엔 비밀번호 원문이 아니라 해시값(bcrypt)만 들어갑니다. 로그인 시 입력값을 같은 방식으로 해시해 비교합니다. 평문 저장은 유출 한 번에 전 직원 계정이 그대로 뚫리므로, 자체 관리를 택한 이상 이 원칙이 가장 중요합니다.
왜 Supabase Auth를 안 쓰나. Supabase 기본 인증에 묶으면 편하지만, 나중에 사내 SSO로 옮길 때 계정 데이터를 통째로 이전해야 합니다. 자체 users 테이블로 두면 SSO는 컬럼 추가로 흡수되고, 기존 계정은 그대로 남습니다. AI Radar가 이미 이 방식(JWT·bcrypt·승인제)으로 구현돼 있어, 인증은 새로 짜지 않고 그대로 재사용합니다 — hub·admin에만 걸고 외부는 공개로 분리.
05 — 외부 · 오퍼링 포탈

고객의 동선 — 스스로 답을 찾게

외부 포탈은 고객이 혼자 읽어도 이해되는 흐름이어야 합니다. 세일즈가 옆에서 설명하지 않아도, 진단부터 리드까지 스크롤 한 번으로 이어집니다. 지금 Offering_portal.html의 9개 섹션이 이미 이 순서입니다.

고객 동선 — 결과를 먼저 보여주고, 연락처는 마지막에. "결과 먼저 → 리드 나중" 원칙.
솔루션 카탈로그는 내부 전용 — 외부엔 최소 정보만. AI Radar의 솔루션 상세(계층·기능·MZC 의견)엔 "우리가 어떤 ISV를 어떻게 보는지"가 담겨, 벤더·경쟁사에 노출되면 안 됩니다. 외부(offering.)엔 오퍼링·진단·리드만 나가고, 카탈로그 자체는 내부에만 보입니다.

06 — 내부 · 인에이블먼트 허브

딜을 골라 들어가고, 업무 순서대로 흐른다

지금 허브는 도구 7종이 그리드에 평평하게 놓인 카탈로그입니다 — PPT를 그대로 옮긴 구조죠. 실제로 직원은 도구를 고르지 않습니다. 딜 하나를 골라 들어가면, 그 딜이 업무 순서를 따라 흐릅니다. 한 번 문 데이터는 끝까지 따라다닙니다.

먼저 — 딜 목록에서 고른다

허브에 들어오면 도구가 아니라 딜 목록이 먼저 뜹니다. 시간이 지나면 여기에 딜이 쌓여, 그 자체가 파이프라인 현황판이 됩니다.

딜 목록 — 고객사 · 현재 단계(①~⑤) · 유형(트랙) · 담당 · 수정일. 포탈에서 막 들어온 D社는 파란색 "신규"로 뜹니다.
"트랙"은 딜 유형입니다. 진단(기존 보안 인프라 + 투자 여력)으로 딜이 4가지로 갈립니다 — T-A 인프라 동반·딜 최대 / T-B 경량·딜 소 / T-C Zscaler 보유·분기 / T-D 타사 SWG·충돌 검토. 목록에서 유형만 봐도 딜 성격과 크기가 짐작됩니다. 내부에서 익숙한 다른 이름(예: 딜 유형·케이스)으로 바꿔도 됩니다.

그 다음 — 딜을 물고 업무 순서대로 흐른다

딜을 고르면 그 딜의 작업 공간으로 들어가고, ①에서 ⑤까지 한 방향으로 이어집니다. 각 단계는 앞 단계의 결과를 물고 시작합니다 — 다시 입력하지 않습니다.

딜 작업 공간 — ①에서 ⑤까지 한 방향. 각 단계가 앞 결과를 물고 시작하니, 같은 걸 다시 입력하지 않습니다.
기술지원 도구는 파이프라인 옆에. 레퍼런스 아키텍처·ISV 계층 기능·인증 트랙은 순서에 묶이지 않는 참조 자료라, 파이프라인과 별도로 언제든 열 수 있는 자리에 둡니다. ②·③에서 기술 검증이 필요하면 거기서 펼칩니다.

딜은 세 경로로 생긴다

자동 · 포탈 유입

고객이 직접

외부 포탈에서 진단·리드 → 딜 목록에 "신규"로 자동 등장. 진단값이 ①에 채워진 채 시작.

수동 · 직접 생성

직원이 시작

전화·미팅으로 들어온 딜. "새 딜 시작"으로 빈 딜 생성 후 진단 입력.

수동 · 시트 회수

진단 시트 배포

포탈에 못 들어오는 고객사엔 진단을 엑셀로 내보내 배포 → 회신 받아 입력.

진단은 하나, 형식은 둘. 지금 구글 시트로 운영하는 21항목 진단(FQA)을 허브 안으로 흡수합니다 — 정답 데이터는 DB에 두고, 고객 배포가 필요할 때만 같은 21항목을 엑셀로 내보냅니다. 시트와 화면이 같은 항목·가중치·임계값(A≥3.5·B≥3.0·C≥3.0·D≥3.5)을 쓰므로, 어느 경로로 들어와도 같은 딜이 됩니다.
07 — 데이터

Supabase — 한 DB, 권한으로 가른다

외부·내부가 같은 테이블을 쓰되, 행 수준 보안(RLS)으로 누가 무엇을 보고 쓸지 정합니다. 이번 개정의 핵심은 deals 테이블 — 딜 하나가 한 행이고, 파이프라인 ①~⑤의 모든 데이터가 여기 묶여 따라다닙니다. 서비스 카탈로그는 AI Radar의 solutions 테이블을 재사용하고(급·포컬·기술제약 컬럼 추가), 회원은 AI Radar의 profiles를 씁니다.

테이블담는 것외부내부
servicesISV·솔루션 (급·상태·포컬)이름·설명만급·포컬·공수 전부
packages서비스 패키지 6종 세부구성·성과물+ 규모급·MD
fqa_items진단 21항목·가중치읽기읽기+판정
tracks트랙 판정 문구고객용 문구내부 경고 포함
focal_contacts포컬 세일즈 명단접근 불가전체
deals ★딜 하나 = 한 행. 고객·진단 21점수·트랙·ISV 조합·패키지·현재 단계·담당접근 불가읽기·쓰기
leads외부 상담 요청 → deals로 승격쓰기 전용읽기
audit_log변경 이력접근 불가관리자
users직원 계정 · password_hash(해시만) · role · SSO 컬럼은 나중접근 불가관리자만
상태(status) 컬럼이 핵심. active / paused / draft로 서비스를 코드 수정 없이 켜고 끕니다. AI Guard처럼 announce 전인 건 paused로 둬서 내부엔 "준비중"으로 보이되 외부엔 안 나오게 합니다.
deals가 파이프라인을 실어 나릅니다. 딜 하나에 — 고객 정보(포탈에서 온 것) · 진단 21항목 점수 · 트랙 판정 · 선택한 ISV 조합 · 패키지·공수 · stage(①~⑤ 현재 단계) · 담당자 · 이력이 다 담깁니다. 직원이 딜을 열면 이 데이터가 통째로 따라오니 단계마다 다시 입력할 필요가 없습니다. 그리고 딜이 쌓이면 — 나중에 "어떤 조합이 실제로 성사됐나" 분석까지 열립니다. 확장성의 최종 목적지입니다.
08 — 협업 · 알림

바꾸면 즉시 저장되고, 슬랙으로 알린다

영업 직원이 딜을 진행하며 내용을 바꾸면, 별도 저장 버튼 없이 입력하는 대로 DB에 들어갑니다. 그리고 딜 상태가 바뀌면 슬랙 채널에 자동으로 알림이 뜹니다 — 팀이 허브에 안 들어가도 딜 흐름을 놓치지 않습니다.

담당자가 바꾸면 즉시 저장 → 다른 팀원 화면은 실시간 갱신(읽기), 슬랙엔 상태 알림. 쓰기는 담당자만.
실시간 저장

바꾸면 바로 DB

  • 자동 저장 — 입력·상태 변경이 즉시 deals에 반영
  • 실시간 동기화 — 다른 팀원 목록도 즉시 갱신 (Supabase Realtime)
  • 담당자만 쓰기 — 딜당 편집자 1명, 나머지는 읽기. 동시 편집 충돌 원천 차단
슬랙 알림

상태 바뀌면 채널로

  • 신규 딜 — 포탈 유입 시 "새 딜, 담당 미배정"
  • 단계 이동 — ①→⑤ 옮길 때 누가·어디로
  • 담당 배정 — 미배정 딜을 누가 맡았는지
  • 범위 알림만(A). 슬랙에서 조회·입력은 후순위
담당자만 저장 — 오히려 단순해집니다. 딜당 편집자를 한 명으로 잠그면 "두 사람이 같은 딜을 동시에 고쳐 충돌"이 아예 안 생깁니다. 실시간 동기화는 "남이 바꾼 걸 즉시 본다(읽기)"에만 쓰이고, 쓰기는 담당자 한 명. 담당을 넘기려면 명시적으로 재배정합니다.
09 — 관리자

DB에 직접 쓰는 유일한 곳

확장성의 핵심이자, 원본 데이터를 바꿀 수 있는 단 하나의 접근점입니다. 외부·내부 사이트는 데이터를 읽기만 하고, 실제 추가·수정·삭제는 전부 여기서 일어납니다. 세일즈·기획이 개발자 없이 직접 합니다.

추가 · CREATE

새 ISV·패키지·포컬을 DB에 직접

폼 입력 → 저장 → 외부·내부 양쪽에 즉시 반영. 재배포 없음. 새 SWG(Netskope 등)가 나와도 코드를 안 건드립니다.

중지 · UPDATE status

토글 하나로 켜고 끄기

삭제가 아니라 paused. AI Guard처럼 announce 전인 건 숨겨두고, 나오면 다시 켭니다.

수정 · UPDATE

급·포컬·설명 편집

포컬 배정(D3)이 나면 이름을 넣습니다 → 딜 설계기의 "포컬 연결"에 바로 반영. MD 요율(D2)도 여기 한 줄.

삭제 · SOFT DELETE

지워도 복구 가능

실제로는 status=deleted. 실수로 지워도 되돌립니다. 물리 삭제는 관리자도 별도 확인 후.

구현 방식 — 두 단계. 1차는 Supabase가 기본 제공하는 테이블 편집기로 시작합니다 (별도 개발 없이 바로 DB 직접 편집 가능). 운영이 익으면 2차로 ISSU 전용 관리자 화면을 얹어, 급·포컬처럼 자주 바꾸는 것만 폼으로 단순화합니다. 모든 쓰기는 audit_log에 기록되어 누가 언제 무엇을 바꿨는지 남습니다.

서비스에 넣는 기본 정보 — HALO 회신이 곧 입력값

관리자가 서비스를 추가·수정할 때 채우는 필드는, HALO 확인 요청서의 회신 항목 그대로입니다. HALO가 시트에 답을 주면 관리자가 그 값을 서비스에 옮겨 담고, 그 순간 외부·내부 양쪽에 반영됩니다.

산출물 · 배포된 두 사이트

향후 계획 — ISV 추천 고도화

지금 ISV 추천은 진단 결과(FQA 미달 항목)가 필요한 솔루션을 지목하는 규칙 기반입니다. 데이터가 없어도 항목별로 근거를 설명할 수 있다는 게 장점입니다. 딜 파이프라인이 돌며 deals 데이터가 쌓이면, "유사 딜에서 성사된 조합" 같은 데이터 기반 추천을 더해 정확도를 높입니다.

머신러닝 추천은 데이터가 모인 뒤에. 학습형 추천 시스템은 과거 딜이 충분히 쌓여야 의미가 있습니다. 지금은 모수가 없어 규칙 기반으로 시작하고 — 이 도구를 쓰는 것 자체가 나중의 추천을 위한 데이터를 모으는 일입니다. 규칙(설명 가능) → 집계(유사 딜 성사율) → 학습(pg_vector 유사도)의 순서로, 데이터가 차오르는 만큼 단계적으로 올립니다. 매출로 이어지는 추천이라 "왜 이걸 추천했나"를 설명할 수 있는 것이 계속 우선입니다.
승인 요청. 이 기획에서 확인받고 싶은 것 여섯 — ① 외부·내부·관리자 3접근 레벨 분리 ② 서브도메인 URL 분리(offering. / hub. / admin.) ③ 내부를 딜 파이프라인(딜 선택 → ①~⑤)으로 재편 ④ Supabase 단일 DB + 권한 분리 ⑤ 관리자가 DB에 직접 쓰는 유일한 접근점 ⑥ 실시간 저장(담당자만) + 슬랙 알림(A) ⑦ 이메일·비밀번호 인증 + 자체 users 테이블(SSO는 나중). 이 일곱이 정해지면 2단계 시안으로 갑니다.
